# Terraform/OpenTofu companion action: product brief

## Status

This is an initial design direction, not a released interface. The companion belongs in a separate repository and release stream. `setup-aws-credential-helper` remains narrowly responsible for renewable AWS profiles and does not become a Terraform wrapper.

## Problem

Using multiple AWS identities correctly with Terraform or OpenTofu requires coordinating independent consumers:

- the S3 backend authenticates in Terraform/OpenTofu core;
- each AWS provider instance authenticates in a provider plugin;
- backend configuration cannot consume provider configuration;
- generated credential-helper paths are private and valid only in the current job;
- saved plans and state-related files are sensitive and have lifecycle/integrity constraints.

The companion should remove repetitive workflow glue without hiding these boundaries or turning arbitrary unreviewed input into privileged command execution.

## Composition model

The preferred workflow is explicit two-step composition:

```yaml
- uses: vegardx/setup-aws-credential-helper@<sha>
  with:
    profiles: ...
    default-profile: deployment

- uses: vegardx/<companion-action>@<sha>
  with:
    engine: terraform
    operation: plan
    working-directory: infrastructure
    backend-profile: state
```

The companion requires the credential action's generated environment:

```text
AWS_CONFIG_FILE
AWS_PROFILE
AWS_SDK_LOAD_CONFIG=1
```

It must validate that `AWS_CONFIG_FILE` is an absolute private regular file and that the requested backend profile exists, but it must not parse, copy, print, upload, or take ownership of credential metadata/cache contents.

## Proposed v1 responsibility

The companion may:

- select `terraform` or `tofu` from an already-installed executable;
- validate a repository-relative working directory;
- generate a mode-`0600`, job-local backend override containing:
  - `profile = "<backend-profile>"`;
  - `shared_config_files = ["<AWS_CONFIG_FILE>"]`;
- run `init` followed by exactly one explicit operation: `plan`, `apply`, or `destroy`;
- pass structured, narrowly scoped arguments to the engine;
- emit non-secret paths and machine-readable outcome metadata;
- remove its generated backend file and temporary files in a post step.

The companion must not:

- duplicate GitHub OIDC or STS logic;
- accept raw AWS access keys;
- synthesize provider blocks or edit `.tf` files;
- infer provider aliases from role names;
- create IAM roles, OIDC providers, state buckets, or lock tables;
- upload plans automatically;
- transfer plans, generated config, or credentials across jobs;
- run arbitrary shell snippets before or after Terraform;
- silently auto-approve destructive operations.

## Provider and backend profiles

`backend-profile` selects credentials only for the S3 backend. Provider identities remain Terraform/OpenTofu configuration, for example:

```hcl
provider "aws" {
  profile = "deployment"
}

provider "aws" {
  alias   = "dns"
  profile = "dns"
}
```

This preserves Terraform's provider model and keeps role-to-provider mapping reviewable in HCL. The companion should not offer a JSON input that rewrites or generates provider aliases: that would be a second configuration language, obscure module provider passing, and make plans harder to audit.

## Candidate v1 inputs

| Input | Required | Direction |
| --- | --- | --- |
| `engine` | no | `terraform` or `opentofu`; default should be chosen deliberately before release |
| `operation` | yes | Exactly `plan`, `apply`, or `destroy` |
| `working-directory` | no | Repository-relative directory; default `.` |
| `backend-profile` | yes for S3 backend mode | Existing generated AWS profile name |
| `backend-config` | no | Structured JSON/YAML object or path to reviewed HCL; merge with generated credential fields without logging values |
| `var-file` | no | Repeatable repository-relative reviewed paths |
| `variables` | no | Prefer environment-based `TF_VAR_*` owned by the workflow; avoid secret-bearing command lines |
| `plan-path` | operation-dependent | Private job-local path; generated by `plan`, required by saved-plan `apply` |
| `lock-timeout` | no | Validated Terraform duration, passed to supported operations |
| `parallelism` | no | Bounded positive integer |
| `auto-approve` | no | Default false; explicit opt-in for apply/destroy |

Avoid a free-form `args` string in v1. If escape-hatch arguments are unavoidable, use an array with denylisted flags that would override backend/config/data-directory safety boundaries, and document that the caller owns their semantics.

## Operation semantics

### Plan

1. Validate environment, engine, paths, profile, and repository state assumptions.
2. Write private generated backend configuration beneath `RUNNER_TEMP`.
3. Run `init -input=false` with that backend config.
4. Run `plan -input=false -out=<private path>`.
5. Optionally run `show -json` only when explicitly requested; treat output as sensitive.
6. Return the plan path for later steps in the **same job**.

### Apply

Support two explicit modes:

- apply a saved plan from the same job; or
- direct apply of configuration with explicit `auto-approve` opt-in.

Do not accept a plan artifact from another job in v1. Safe cross-job plan transfer requires a separate design for confidentiality, integrity, provenance, exact source/provider/module lock identity, engine version, working-directory contents, and approval timing.

### Destroy

Require `operation: destroy` plus explicit `auto-approve: true`. GitHub Environments, reviewers, concurrency, and deployment authorization remain workflow responsibilities. The action should print a clear operation summary without exposing variables or plan content.

## Tool installation boundary

Prefer an already-installed engine in v1 and report its exact version. Tool downloading, checksum/signature verification, mirrors, credentials, and version resolution are substantial supply-chain responsibilities. They can be provided later by a separate setup action or a tightly pinned installer mode; they should not be casually bundled into execution.

The companion should validate supported version ranges and fail clearly rather than silently selecting or upgrading a binary.

## Files and environment

Use a private temporary root and set a private `TF_DATA_DIR` so generated backend metadata does not pollute or race with the checkout. Preserve Terraform/OpenTofu's normal plugin and module lock behavior; never rewrite `.terraform.lock.hcl` without an explicit mode.

Potential safe outputs:

```text
engine
engine-version
operation
working-directory
plan-path             # same-job use; path only
exit-code
has-changes           # from detailed-exitcode for plan
```

Never output credential values, generated AWS config contents, backend config contents, plan JSON, state, or sensitive variables.

## Plan and artifact security

A saved plan can contain sensitive values and executable deployment intent. Applying it later also depends on exact source, provider/module selections, variables, backend, engine version, and trust context.

Therefore v1 should:

- keep plans private and same-job by default;
- never upload/download artifacts automatically;
- use mode `0600` where supported;
- avoid printing `show -json` output;
- document that GitHub artifact encryption-at-rest does not by itself establish trusted provenance or appropriate readership;
- leave cross-job plan/apply workflows for a later, explicit threat-modelled feature.

## Concurrency and state safety

The action should expose `lock-timeout` but must not replace backend locking. Workflow-level `concurrency` should be recommended per environment/state identity to prevent overlapping deployments before they contend on the backend.

The action must not force-unlock automatically. A lock failure should preserve diagnostics and require operator review.

## Testing direction

The companion's acceptance suite should compose with the released credential helper and cover both engines:

- independent `state`, default provider, and aliased provider profiles;
- backend initialization without static credentials;
- plan no-change/change exit semantics;
- saved-plan apply in the same job;
- direct apply and destroy opt-in controls;
- provider and backend renewal during one long operation;
- paths containing spaces;
- failure cleanup without masking the primary failure;
- rejection of untrusted/absolute traversal paths and conflicting AWS credentials;
- no credential, backend config, state, or plan content in logs.

Live AWS testing remains owner-run and least-privilege. Moto can validate plumbing and process behavior but not IAM, trust policy, or real expired-token enforcement.

## Decisions to make before implementation

1. Product/repository name and whether both engines have equal support from v1.
2. Whether v1 supports only preinstalled engines or integrates with a dedicated installer.
3. Backend scope: S3-only initially, or a generic reviewed backend HCL path with S3 credential augmentation.
4. Whether `plan`, `apply`, and `destroy` live in one action entrypoint or separate narrowly permissioned actions.
5. Exact same-job saved-plan contract and how source/config drift is detected.
6. Whether provider profile validation is informational only or omitted entirely; provider mapping should remain in HCL.
